# Поддержка ключей i18n (react-i18next) — дизайн

Дата: 2026-06-09
Плагин: CSS Modules Scoped Usages (далее расширяем тем же плагином)

## Проблема

В проекте `WebDX/web` используется `react-i18next`. Ключи переводов — это
точечные пути в большой вложенный JSON (`src/lang/translations/en.json`,
~6965 строк, ~375 КБ). Из-за размера и глубины написать типы на TypeScript
нереально, поэтому в `t('...')` нет ни автокомплита, ни проверки ключей.

Цель: дать на уровне IDE
1. **автокомплит** ключей внутри `t('...')`;
2. **инспекцию** «неизвестный ключ» (подсветка опечаток/несуществующих ключей);
3. **go-to-definition** (переход из ключа к месту в `en.json`) + обратный
   find-usages.

## Контекст проекта (факты)

- Один namespace `translation` — каждый ключ это точечный путь в локали
  (`page.pwa_install.title`, `common.action.copy`).
- `useTranslation()` всегда вызывается **без аргументов** (все 811 мест).
- Источник ключей — `en.json`; рядом ещё 8 локалей. Интерполяция в значениях
  через `{{name}}` (на ключи не влияет).
- Есть **динамические** ключи через template literal
  (`` t(`page.pwa_install.steps.${index+1}`) ``) — статически не резолвятся.
- Точки вызова: `t('...')` из `useTranslation()`, отдельный `t` из
  `initLocales.ts`, `i18next.t(...)` (9×), `i18n.t(...)` (1×),
  `<Trans i18nKey="...">` (132 файла).

## Решение (выбранный подход: reference-centric)

Все три фичи читают один общий индекс. Поверх него — один `PsiReference`
(навигация + find-usages + автокомплит) и отдельная инспекция (подсветка).

### 1. `I18nKeyIndex` — общий индекс (project service)

- **Поиск JSON (по конфигу):** найти файл, который настраивает i18n — импортит
  `initReactI18next` из `react-i18next` и/или вызывает `.init({ resources … })`.
  В нём пройти по ES6-импорту с локальным именем `en` к его `*.json`. Это и есть
  источник ключей.
  **Fallback** (если детект не сработал): конвенция
  `**/lang/translations/en.json` → `**/translations/en.json` → любой `en.json`.
  Лучшее-из-возможного: если файл не найден — фичи молча выключаются (не краснит
  весь проект).
- **Извлечение ключей:** обойти JSON PSI; каждый **строковый лист** даёт точечный
  путь. Объекты-ветки ключами не являются. Результат: `Set<String>` валидных
  ключей.
- **Навигация:** dot-path → `JsonProperty` спуском по дереву объектов on-demand
  (без хранимых указателей, чтобы не протухали). Разделитель сегментов — `.`
  (дефолт i18next; сегменты тут snake/camelCase, точек не содержат).
- **Кэш:** `CachedValuesManager` с зависимостью от JSON-файла (пересборка только
  при изменении `en.json`); найденный файл кэшируется относительно init-файла.

### 2. `I18nCallSites` — общий матчер (util)

Один предикат для reference-contributor и инспекции (правила матчинга в одном
месте). По строковому литералу отвечает: «это ключ i18n, и какой у него текст?».
Матчит:
- первый аргумент вызова, у которого callee — `t`, `i18next.t` или `i18n.t`;
- значение атрибута `i18nKey` у `<Trans>`.

**Гарды:** только **plain string literal** (template literal с `${…}` —
пропускаем, и для автокомплита, и для инспекции); только в файлах, которые
импортят `react-i18next`/`i18next` (убирает ложные срабатывания на чужих
функциях с именем `t`). Переименованные биндинги не поддерживаем.

### 3. `I18nKeyReference` (через `PsiReferenceContributor`)

Вешается на сматченный литерал. `resolve()` → `JsonProperty` (go-to-definition
+ двусторонний find-usages бесплатно). `getVariants()` → все ключи как
lookup-элементы (английское значение в tail text) — это и есть автокомплит.
- *Заметка по реализации:* если платформа не отдаёт `getVariants()` как
  автокомплит для JS-литералов, добавлю тонкий `CompletionContributor`,
  делегирующий в тот же индекс (те же данные, без дублирования логики).

### 4. `I18nUnknownKeyInspection` (LocalInspectionTool)

Использует `I18nCallSites`. Если сматченный plain-string ключ не в индексе →
**WARNING** «Unknown translation key '…'». Пропускает динамические ключи; молчит,
если файл ключей не найден. Quick-fix в этой итерации нет.

### 5. Упаковка и тесты

- Новый пакет `com.webdx.i18n`, регистрация в `plugin.xml`
  (`psi.referenceContributor`, `localInspection`, project service). Существующий
  CSS-код не трогаем. Bump версии.
- Имя плагина пока остаётся «CSS Modules Scoped Usages» (id не меняем); по факту
  плагин уже шире — переименуем позже при желании.
- **Тесты (TDD, как в существующем сьюте):**
  - чистая логика: flatten JSON → set ключей; предикат точки вызова; правило
    детекта `en`-импорта;
  - функциональные `BasePlatformTestCase`: автокомплит предлагает ключ; инспекция
    краснит неизвестный / молчит на известном / игнорит динамический; reference
    резолвится в `JsonProperty`; `<Trans i18nKey>`; негативный гард (нет
    i18n-импорта → нет reference/инспекции).

## Не входит в объём (YAGNI)

- Hover-превью значения и проверка пропущенных переводов в других локалях.
- Поддержка переименованных биндингов и кастомных namespace.
- Quick-fix «создать ключ».

## Прирост: проверка интерполяции (добавлено 2026-06-09)

Для `t('key', { … })` значение ключа (`{{placeholder}}`) задаёт ожидаемые
переменные.

- **Парсинг плейсхолдеров** — `I18nKeys.placeholdersOf(value)` (регекс `{{ x }}`,
  тримит пробелы, для `{{x, fmt}}` берёт имя до запятой).
  `I18nKeyIndex.placeholders(project, key)` достаёт значение ключа и парсит.
- **defaultVariables из конфига** — `I18nConfig.defaultVariableNames(initText)`
  вытаскивает ключи из `interpolation.defaultVariables { … }`
  (`specialistsCount`, `specialistsCountFull`); они никогда не обязательны.
- **`I18nOptions`** — находит `t(...)`-вызов и его 2-й аргумент (только call-форма,
  не `<Trans>`); allowlist зарезервированных опций i18next (`count`, `context`,
  `defaultValue`, …).
- **`I18nInterpolationInspection`** (WARNING): неизвестный ключ объекта (опечатка),
  пропущенный обязательный плейсхолдер, или вовсе нет объекта опций; `...spread` /
  вычисляемый ключ выключают проверку «пропущено».
- **`I18nOptionsCompletion`**: автокомплит имён плейсхолдеров внутри
  `t('key', { <caret> })` (исключая уже введённые).
- Тесты: `I18nPlaceholdersLogicTest`, `I18nDefaultVariablesLogicTest`,
  `I18nInterpolationIndexTest`, `I18nInterpolationInspectionTest`,
  `I18nOptionsCompletionTest`. Итого 90 тестов.
