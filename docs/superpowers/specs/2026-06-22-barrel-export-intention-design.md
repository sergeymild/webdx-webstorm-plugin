# Дизайн: Alt+Enter — экспорт компонента через barrel-модули вверх по дереву

**Дата:** 2026-06-22
**Статус:** Утверждён (дизайн)
**Компонент:** `com.webdx.barrels` (новый)

## Проблема

Во многих React/TS-проектах компоненты публикуются наружу через «barrel»-файлы
`index.ts(x)` (`export … from`). Когда разработчик создаёт новый компонент, его
нужно вручную прописать в цепочку barrel'ов от папки компонента вверх до публичного
API модуля — иначе автоимпорт в местах использования не предложит правильный путь
(через barrel), и импорт придётся писать руками на файл напрямую.

Хочется: поставить каретку на имя компонента, нажать **Alt+Enter (Option+Enter)** и
одним действием прописать re-export во все существующие `index.ts(x)` вверх по дереву —
до логической границы модуля.

Реальные обследованные проекты (подтверждают, что устройство barrel'ов различается, и
эвристика должна работать на обоих):

- **`intch/web`** — один пакет, tsconfig `"@/*": ["./*"]` (алиас на корень проекта).
  Barrel'ы на **каждом уровне** (лист-папка → промежуточный → модульный). «Корни
  модулей» (`UI-KIT/`, `components/`, `src/containers/`) — это вершина barrel-цепочки:
  выше них (`src/`, корень) `index` нет. Стиль: одинарные кавычки, **без `;`**,
  преимущественно `export * from './X'`, дефолтный экспорт — `export { default as X } from`.
  Пример разрыва: `src/containers/TouchpointCreateContainer/index.ts` =
  `export { default } from './TouchpointCreateContainer'`, но в `src/containers/index.ts`
  он не прописан.
- **`react-native-musescore`** — один пакет, 118 tsconfig path-алиасов, каждый указывает
  прямо на `index.ts` модуля (`@mu-native/score-screen` → `./src/sections/score/screens/score/index.ts`).
  Barrel **только** в корне модуля; промежуточных `index` нет → лист реэкспортится одним
  модульным `index` через многосегментный путь (`./components/modals/RatingScoreSheet`).
  Стиль: одинарные кавычки, **с `;`**, преимущественно именованные `export { X } from`.

## Scope

В scope: `intentionAction`, который по Alt+Enter на имени экспортируемого top-level
объявления дописывает `export … from` **только в существующие** barrel-файлы вверх по
дереву, до автоматически вычисленной границы модуля.

**Не трогает файлы-потребители** — правильные импорты там появятся сами через автоимпорт
IDE после построения barrel-цепочки. **Не создаёт** отсутствующие `index`-файлы (их
разработчик создаёт сам). Подтверждённые границы scope — см. «Out of scope».

## Архитектура

Новый пакет `src/main/kotlin/com/webdx/barrels/`, по аналогии с остальными.

- **`BarrelExports.kt`** — `object` со всей чистой/PSI-логикой (полностью тестируемой):
  вычисление source-root cap и границы модуля, сбор цепочки barrel-уровней, вычисление
  относительного пути и формы re-export для каждого уровня, дедуп уже прописанных уровней.
  Без зависимостей от UI.
- **`BarrelExportIntention.kt`** — тонкий `PsiElementBaseIntentionAction`. `isAvailable`
  (гейт + вычисление корня модуля для текста), `invoke` (применить план в
  `WriteCommandAction`). Логики не содержит — делегирует в `BarrelExports`.
- **`plugin.xml`** — одна регистрация `<intentionAction>` (категория `JavaScript`), без
  привязки к языку (как `CssModuleImportSymbolIntention`); доступность решает `isAvailable`.

Переиспользуем существующее: `CssModules.tsconfigAliases(project)` — для резолва
`@/` / `baseUrl` → source-root и для распознавания каталогов-целей алиасов; ES6-PSI
(`ES6ExportDeclaration`, `ES6ImportHandler.isExported`) — для определения экспортируемого
символа под кареткой; паттерн прямой вставки в документ из `CssModuleImportSymbolIntention`.

## Определение границы модуля и цепочки

`BarrelExports.planFor(componentFile, symbolName, isDefault, project): Plan?` строит план:

1. **Source-root cap** = каталог, на который указывает `@/` / `baseUrl` (через
   `tsconfigAliases`); если нет — каталог ближайшего `package.json` (корень репы). Выше cap
   обход не идёт никогда.
2. **Обход вверх** от каталога компонента до cap включительно. Для каждого каталога-предка
   `D`: если в `D` есть `index.ts(x)` — это barrel-уровень. Отсутствующие `index`
   пропускаем и идём выше.
3. **Корень модуля** (стоп включительно) — первый при подъёме каталог, который:
   - содержит `package.json` (workspace-пакет монорепы), **или**
   - его `index` — цель tsconfig path-алиаса (кейс musescore), **иначе**
   - самый верхний barrel-уровень, найденный до cap (кейс intch — упрётся в
     `components`/`UI-KIT`/`src/containers`).
4. **Результат** — упорядоченный список barrel-уровней от каталога компонента до корня
   модуля включительно, плюс сам корень модуля (для текста intention).

Примеры:

- intch `src/containers/TouchpointCreateContainer/TouchpointCreateContainer.tsx`
  (default-export): уровни = `TouchpointCreateContainer/index.ts` (есть) →
  `src/containers/index.ts` (есть); выше — `src/` без index, `@/`→корень. Корень модуля =
  `src/containers`.
- musescore лист `src/sections/score/screens/score/components/modals/RatingScoreSheet.tsx`:
  единственный barrel-уровень `screens/score/index.tsx` (он же алиас-цель
  `@mu-native/score-screen`) → корень модуля.

## Путь и форма re-export

Имя, под которым символ станет доступен у корня модуля, — `X`: имя named-экспорта; для
default-экспорта — объявленное имя, а для анонимного `export default` — basename файла.

Для каждого barrel-уровня `B` (каталог `dirB`) определяем **цель `T`** и **строку**:

- `T` = ближайший существующий barrel-каталог *ниже* `B` на пути к компоненту; если такого
  нет — сам **файл компонента** (относительный путь от `dirB`; при пропуске не-barrel
  каталогов путь многосегментный; расширение файла отбрасывается).
- **Форма:**
  - `T` — файл компонента, символ **default** → `export { default as X } from 'T'`
    (`export *` не пробрасывает `default` — это обязательное правило корректности).
  - `T` — файл компонента, символ **named** → `export * from 'T'`, если в этом `index`
    доминирует звёздный стиль, иначе `export { X } from 'T'`.
  - `T` — дочерний barrel-каталог → `export * from './child'` (звезда несёт named `X`)
    либо `export { X } from './child'` — по доминирующему стилю файла.
- **Конверсия default→named** происходит ровно на нижнем уровне, указывающем на файл; выше
  `X` уже именованный, поэтому работают и звезда, и именованный re-export.

**Стиль берётся из самого целевого `index`-файла:** вид кавычек (одинарные/двойные),
наличие `;`, преобладание `export *` vs именованного re-export. Если в файле нет таких
маркеров (пустой) — дефолт: одинарные кавычки, без `;`, по доминирующему стилю проекта,
выведенному из других barrel'ов. Новая строка дописывается в конец файла.

## Поведение intention

- **`isAvailable`** = true, когда:
  - файл — `.ts/.tsx/.js/.jsx` и **не** сам `index.*`;
  - каретка на имени **экспортируемого** top-level объявления (`export const/function/class`,
    `export default`, локальный `export { X }`);
  - есть хотя бы один barrel-уровень-предок;
  - **хотя бы один уровень ещё не форвардит `X`** (плану есть что дописать).

  Иначе intention не предлагается.
- **Дедуп.** Уровень считается «уже форвардящим `X`», если в нём есть re-export, делающий
  `X` достижимым по имени: именованный `X`, `export *` от дочернего barrel, который уже
  отдаёт `X`, либо (для default) `export { default as X }`. Такие уровни пропускаются,
  строки не дублируются.
- **`getText`** динамичный: «Export `X` through barrels up to `<moduleRoot>`». Корень
  вычисляется в `isAvailable` и сохраняется в поле инстанса (IDE зовёт `isAvailable` →
  `getText` на одном инстансе). **`getFamilyName`** статичный: «Export through barrel modules».
- **`invoke`** применяет план: правит все целевые `index`-документы в одном
  `WriteCommandAction` (один Ctrl+Z откатывает всё), затем `commitDocument`. Резолв всегда
  против `containingFile.originalFile` (gotcha #8).
- Если символ **не экспортирован** — intention не предлагается (требуем `export`).
  Авто-добавление `export` — вне scope.

## Обработка ошибок и краевые случаи

Принцип: всегда безопасно — лучше не предложить intention, чем сломать код.

- Нет `tsconfig`/алиасов → cap = каталог ближайшего `package.json`; нет и его → самый
  верхний найденный barrel-уровень.
- Несколько `index.*` в одном каталоге (`.ts`/`.tsx`/`.js`) → берём существующий; при
  нескольких предпочитаем `.ts`/`.tsx`. Пишем в один.
- Анонимный `export default` → имя `X` из basename файла; каретку ловим на ключевом слове
  `default`.
- Все уровни уже форвардят `X` → intention не предлагается.
- Коллизия имён (два соседних файла экспортят `X` через `export *` в одном barrel) — не
  разбираем: дописываем строку, TS сам предупредит. Документированное ограничение.
- Всё PSI-чтение — внутри read-action; запись — в `WriteCommandAction`; коммит документов
  после.

## Out of scope (YAGNI)

- Правка существующих импортов в файлах-потребителях (подтверждено: фича только дописывает
  barrel'ы).
- Создание отсутствующих `index`-файлов (их создаёт разработчик).
- Авто-добавление `export` к необъявленному-как-export символу.
- Rename / Safe Delete для barrel-связей.
- Не-ESM формы (`module.exports` / CommonJS).
- Разрешение коллизий одноимённых экспортов в одном barrel.

## Тестирование

Как везде: `BasePlatformTestCase`, реальный WebStorm SDK, in-memory fixtures, без моков.
Запуск: `JAVA_HOME=… ./gradlew test`.

- **`BarrelExportsTest`** (логика `BarrelExports`):
  - граница модуля — `package.json`, алиас-цель, «самый верхний index до source-root»;
  - путь — тот же каталог; пропуск не-barrel уровней (многосегментный путь); дочерний
    barrel-каталог;
  - форма — named-звезда vs named; `default as X`; конверсия default→named на нижнем уровне;
  - дедуп — пропуск уже форвардящих уровней; пустой план, когда всё прописано.
- **`BarrelExportIntentionTest`** (доступность + применение через реальный intention):
  - предлагается на экспортируемом имени при наличии barrel-предка; **не** предлагается в
    самом `index`, без barrel-предков, для не-экспортируемого символа, когда всё уже
    прописано;
  - применение даёт корректные строки во всех целевых `index` со стилем файла; кейс
    default-экспорта; всё под одним `WriteCommandAction`.
  - **Две формы-фикстуры под реальные проекты:** intch (barrel на каждом уровне, `@/`→корень,
    без `;`, звёздный стиль) и musescore (один модульный barrel через алиас, многосегментный
    путь, с `;`, именованный стиль).

В отличие от Alt+Enter-квикфикса нерезолвнутого импорта (не тестируется в лёгкой фикстуре,
т.к. не запускается TS-сервис), эта intention работает на исходном PSI + редактировании
документа и **полностью покрывается фикстурами**.

## Регистрация в тестовом рантайме

Никаких новых bundled-плагинов: `JavaScript` (+ его зависимость `com.intellij.modules.json`)
уже подключены в тестовой сборке для остальных JS/TS-фич. Граничные сигналы (`package.json`,
`tsconfig.json`) — обычные файлы в фикстуре, плагины для них не нужны.
